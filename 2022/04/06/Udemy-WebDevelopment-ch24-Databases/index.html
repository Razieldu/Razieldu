資料庫

在23章節的專案,我們發現到只要伺服器重新啟動,首頁上面的文章標題都會消失,沒辦法永久儲存資料,因此我們需要資料庫的協助

資料庫的種類有很多種,但是大致上分為兩類

SQL(Structured Query Language):
It is a really old school piece of technology that businesses have been using for many years(MySQL,Postgres)

A sequel database will group your data into tables(表格),假設今天我們的表格有三個標題,分別為姓,名,地址,但是實際上有些客人僅願意提供電子郵件,或者是僅願意提供姓名,our SQL database will try to square off the table(把缺少的資料部分自動設為null,或是新增加一個標題類別),null can be very dangerous,expecially if you are not expecting something to be null or not have any data

Old and reliable and it likes structure.

Relational databases

SQL databases are really good at establishing relationships between your data,you will group related pieces of data into individual tables,so you might have a customer table,a product table,and an order table.And in your database you can specify a relationship between the tables,so for example you could link up the customers table through the customer ID to the orders table,and you could link the products table to the product ID in the orders table.

NOSQL(Not Only Structured Query Language):
they can be any format as long as it is not SQL although most of them follow a very similar structure(MongoDB,redis)

If we were working with Mongo in the same situation(第一點部分提及),the data would be represented as JSON objects,you will have key value pairs such as first name,last name,address and if one of those documents you had an extra key value pair such as email then is does not affect any of the other data records. And none of the records or documents actually have to be the same shape or follow the same structure

Tend to be flashier ,newer with modern syntax and modern method,flexible

It is really helpful for startups where your data structure is not predefined,you are not bound to a particular structure

Non-Relational databases
It is not quite efficient or fast as a SQL database

根據實際情況來決定使用哪種資料庫

SQL:資料間的關聯性比較強
NOSQL:You hava a single user generating lots of content creating a one user to many post relationship(Instagram)

Scalability(規模可伸縮性)

  1. NOSQL的優勢點
  2. 使用SQL資料庫建立資料,當資料越來越多時,會發現電腦無法負荷,就像在蓋高樓,越蓋越高,有倒塌的可能
  3. 使用NOSQL資料庫建立資料時,就像蓋了很大量但是低樓層但是穩固的建築,所有的資料都分布在各個建築物裡面

SQL

  1. more mature(成熟的技術)
  2. table structure
  3. require a Schema(輪廓;略圖)
  4. great with relationships
  5. scale vertically

NOSQL

  1. shiny and new(持續在更新中)
  2. document structure
  3. more flexible to changes
  4. not great with complex relationships
  5. scale horizontally
