Working Directory:

We create a file in our working directory inside our Story directory.The working directory is the folder or the directory where you initialize your Git Repository.When we say git init,we did that inside the Story directory.And from now on,Git is going to try and track the changes that it sees between the working directory and the local repository.

Staging Area:

We use git add to push the file which we have created in working directory to the staging area.The reason why there is a intermediate staging area: sometimes you might not want to add all of your files to be tracked or all of your files to be committed ,so the staging area is a good place to try and figure out what are the things you want Git to ignore,and what are the things that you want to be tracked.

Local Repository:

We use git commit to push files from staging area to our local repository,so that .git and that version is given a name through the commit message. So,that means even if we messed up our file,we can still use the last version that is under version control and we can use a special command called git checkoutto revert back or roll back to the last position in our local repository

What this line of code does is that it pushes your local repository to the remote repository using the u flag or u option which is basically links up your remote and your local repositories. And then,we are going to push it towards the remote that is called origin and we are going to push it to the branch that is called master

The master branch is simply the default branch or the main branch of all of your commits.

It is going to upload your local repository to the remote repository on GitHub servers and depending on the size of your local repository this can take various amounts of time.

點擊Github上方的insight,點擊Network,可以看到最近有兩個儲存點或者commmits,if we hover over them,you can actually see the commit messages.

We have made a working repository which is the directory which is the directory where we initialized Git. Then we can add our files to the staging area where we can pick and choose which files we want to commit. Then once we are ready then we can commit our files or save safe point to our local repository.So,once you have done a few commits, then you actually have this timeline of various commits,and this is called your master branch.

So,the master branch is your main branch of commits or save points and it is sequential(連續的).The important thing to note is that you can have a local repository completely in parallel(平行下) with a remote repository. Check the difference between them but you can also sync(同步發生) them so that they are the same.

In our case,the local repository is the git file that we have got inside our Story directory and the remote repository is GitHub which hosts our code and also all of the changes that were made in between the different commits.So we have this master branch of various commits in our local Git repository.When we perform the command git push, then that effectively pushed all of those commits,all of those various versions and changes and code pieces to our remote repository on GitHub.

Forking and Pull Requests

Git clone:just grabbing at the entirety of the repository and then cloning it to your local work environment.

Forking:copy a repository that is hosted on GitHub and we are keeping the copy under our GitHub account where we can make changes to it. Once you fork a remote repository ,then yo u now own it. You have full permssions to do whatever it is you like with this copy of repository.

我們去fork一個別人的專案到我們的GitHub上面,之後使用git clone到我們本地的working directory,並在本地進行修改,然後git add ., git commit -m,接著git push到我們的remote repository,假設我們想把我們的修改讓原專案作者也採用,此時我們要向原來的作者make a pull request(提議),假設原作者也覺得我們的提議不錯,原作者會同意我們的pull request,我們的更動會被pull over(決定權不在我們身上,因
此使用pull),原作者進行merge branch ,the changes will be reflected in a new commit

同步remote repository 以及本地

git pull 命令用於從遠程獲取代碼並合併本地的版本。

git pull 其實就是 git fetch 和 git merge FETCH_HEAD 的簡寫。命令格式如下：
git pull <遠程主機名> <遠程分支名>:<本地分支名>

將遠程主機 origin 的 master 分支拉取過來，與本地的 brantest 分支合併。

如果遠程分支是與當前分支合併，則冒號後面的部分可以省略。
