AUTHENTICATION

  1. 步驟 1:Get Access/permission
  2. 步驟 2:Send request to protect resources

We can not just use “yes” or “”no to grant permission because it can easily be faked.

1.Server-Side Sessions(使用在前後端一致的情況)

一旦伺服器驗證了使用者的登入權限,伺服器會儲存一個專屬於該使用者的驗證辨識(identifier),伺服器為了這個請求登入的使用者,產生並儲存專屬的驗證辨識器,也就是說每一個登入的使用者都會有一個專屬的驗證辨識儲存於伺服器,這個驗證辨識不是只有儲存在伺服器,而是也會傳送回去使用者端,也就是說伺服器回傳的 response 也包含了驗證辨識器,所以伺服器端以及使用者端都能夠辨認這個驗證辨識器

客戶端在爾後向伺服器端發送 request 的時候,這個驗證辨識器也會附加在 request 上,也因為伺服器能夠辨認這個驗證辨識器,因此驗證辨識器是無法被偽造的

Create and Store unique identifier on server,send the same identifier to client.

Client sends identifier along with request to protect resources

2.Authentication Tokens(前後端分離)

使用者傳送密碼或是電子郵件信箱到伺服器,伺服器會使用資料庫儲存的資料來進行比對,如果比對後發現是正確的,伺服器會產生一個驗證 token(一個很長的字串,一些編碼完成的資料包含在裡面),伺服器會使用演算法去把資料(密碼以及電子郵件信箱)編碼成一個字串,可以使用解碼取得這些資料,這個驗證 token 是由伺服器使用一個金鑰(is used for hashing the data into a string),並使用特別的演算法產生,並且這個金鑰只有伺服器端掌握

這個驗證 token,不會被儲存,而是會被傳送到客戶端,但是只有伺服器端知道怎麼產生這個 token(因為只有伺服器端知道產生這個 token 的金鑰),客戶端可以使用這個驗證 token 附加在向伺服器發起的 request 上,如此一來就能保護在伺服器上的資料

即使伺服器並沒有儲存這個驗證 token,伺服器還是能夠辨認這個驗證 token 是由伺服器本身所產生的,因為只有伺服器知道產生 token 的金鑰,因此隨著使用者端的 request 而發送到伺服器的驗證 token 能夠被伺服器辨認並驗證通過

Create(not store)”permission token” on server,and send token to client

Client sends identifier along with request to protect resources
